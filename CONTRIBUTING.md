# 贡献指南 / Contributing Guide

感谢你考虑为链上寓言贡献内容。在动笔之前，请阅读以下规范。

## 贡献流程

1. **认领概念**：在 `concepts-catalog.md` 中找到未映射的 wiki-web3 概念，或提交一个新的概念建议。
2. **阅读来源**：仔细阅读对应的 wiki-web3 概念原文，确保理解其技术细节。
3. **选择隐喻**：为你的概念找到一个"世界"。这个世界可以是古代的、现代的、幻想的，甚至是科幻的，但必须满足：
   - 世界内部的运行规则与概念的技术机制一一对应。
   - 世界的冲突点恰好是概念的风险点或设计难点。
4. **写作**：遵循 [templates/fable-template.md](templates/fable-template.md) 的格式。
5. **自检**：运行 `python scripts/validate-fable.py <你的文件>`。
6. **提交 PR**：PR 描述中注明：概念来源、隐喻选择理由、自检结果。

## 叙事质量标准

### 好的寓言是什么样的？

- **有角色，有动机**：角色不只是概念的传声筒。他们有自己的利益和盲点，这些盲点恰好对应技术中的常见误解。
- **有冲突，有代价**：故事中的错误决策必须带来可感知的后果，让读者"痛"到记住这个教训。
- **有层次**：表层是故事，中层是技术映射，深层是对设计哲学的暗示。

### 避免的陷阱

| ❌ 反模式 | ✅ 正确做法 |
|-----------|------------|
| "小明用智能合约存了钱，结果被重入攻击了" | 写一个银行，它的出纳员在对方还没离开柜台时就不断给钱 |
| 把技术术语直接翻译成日常用语（"区块链就是一本公开账本"） | 创造一个世界，这个世界的物理规则恰好复现了区块链的特性 |
| 故事讲完，注解只是重复故事内容 | 注解应该解释"为什么这个故事的设定必须是这样"——即技术约束如何转化为叙事约束 |

## 文件命名规范

```
fables/<category>/XX-the-title-slug.md
```

- `XX`：两位数字序号，按类别内顺序递增。
- `category`：从 `consensus`, `defi`, `security`, `evm`, `cryptography`, `governance`, `infrastructure`, `solana` 中选择。
- `slug`：英文小写，单词间用连字符。

## Frontmatter 规范

```yaml
---
title: "The Infinite Borrower"
title_cn: "无限借贷者"
concept: "Reentrancy Attack"
concept_cn: "重入攻击"
category: "security"
difficulty: "intermediate"  # beginner / intermediate / advanced
author: "your-name"
created: "2026-04-23"
sources:
  - "wiki-web3/concepts/solidity-重入攻击与防御.md"
tags: [security, reentrancy, solidity]
---
```

## 映射表规范

每则寓言必须包含一个 `Key Mappings` 表格，格式如下：

```markdown
| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 金库 | 智能合约 | 存储资金的地方，规则由代码定义 |
| 出纳员 | 提款函数 | 执行资金转移的逻辑单元 |
```

- **故事元素**：读者在故事中遇到的具体事物或角色。
- **技术概念**：对应的 Web3 术语。
- **说明**：为什么这个映射是准确的（不是显而易见的对应才需要说明）。

## 双语文本

- 标题必须双语（`title` + `title_cn`）。
- 正文语言由作者选择。如果正文是中文，鼓励在 PR 中说明是否愿意提供英文版（或反之）。
- 社区可以提交翻译 PR，不需要作者本人双语写作。

## 技术准确性审查

PR 合并前需通过两项审查：

1. **叙事审查**：故事是否完整、吸引人、遵守延迟揭示原则。
2. **技术审查**：映射是否准确，是否存在会误导读者的简化。

如果你提交的是高度专业的概念（如 ZK 证明、MEV 策略），请在 PR 中 `@` 相关领域的贡献者进行技术审查。

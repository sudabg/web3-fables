## 二、工作流

### 单篇寓言的标准工作流

```
Day N (上午): 阅读 wiki 原文 → 选择隐喻世界 → 生成寓言草稿
Day N (下午): 自审叙事质量 → 补充映射表 → 完善注解
Day N (晚上): 运行 validate-fable.py → 修复格式问题 → 提交 PR
Day N+1: 等待审查 / 根据反馈修改 → 合并
```

**每篇实际耗时：4-6 小时（含审查修改）。**

### 并行策略

由于寓言之间没有依赖关系，可以多篇并行创作。但为保证质量，**同一作者同时创作的寓言不超过 2 篇**。

建议的并行方式：
- 按领域分组，每组由一个核心贡献者主导（该领域的技术审查者）
- 组内成员分工：一人主攻故事，一人主攻技术注解

### 审查节奏

- 维护者每日审查一次开放的 PR
- 审查者必须在 48 小时内回复（否则视为弃权，由 maintainer 代审）
- 紧急修复（错别字、链接错误）可走 `hotfix` 分支直接合并

# 评审规范 / Review Guidelines

> 本规范是 PR 合并的硬性门槛。提交者和审阅者都必须逐条对照。

---

## 一、流程：两阶段审查制

每则寓言的 PR 必须通过**两个阶段**的审查才能合并：

| 阶段 | 审查者 | 重点 | 否决权 |
|------|--------|------|--------|
| **叙事审查 (Narrative Review)** | 至少 1 位有寓言写作经验的贡献者 | 故事质量、隐喻选择、延迟揭示 | ✅ 有 |
| **技术审查 (Technical Review)** | 至少 1 位对应领域的开发者 | 映射准确性、概念完整性 | ✅ 有 |

两阶段可以并行，但都必须通过。任一阶段的审阅者挂 `Request changes`，PR 不得合并。

---

## 二、叙事审查清单

审阅者在 PR 中必须逐条回复以下 checklist（复制到 review comment 中勾选）：

```markdown
### 叙事审查

- [ ] **延迟揭示**：前 1/3 篇幅内未出现区块链术语（"智能合约""区块链""Gas"等）。
- [ ] **角色动机**：主要角色有明确的利益诉求和盲点，不只是概念的传声筒。
- [ ] **冲突代价**：故事中存在错误的决策，且该决策带来了可感知的负面后果。
- [ ] **完整性**：故事有开头、发展、高潮、结局，不是一个片段或摘要。
- [ ] **字数**：故事正文（不含注解）在 1500-3000 中文字符（或等效英文词数）。
- [ ] **无代码块**：正文不含 Solidity / Python / JavaScript 等代码块（注解中可以有伪代码或 YAML）。
- [ ] **语言质量**：无明显语病、错别字、标点错误。
```

### 叙事审查的否决标准

以下任一情况，叙事审查者必须 `Request changes`：

1. **过早暴露**：故事前 1/3 明确说出了技术概念（如"这就像区块链上的重入攻击"）。
2. **无角色动机**：角色只是"演示"概念，没有自己的利益、恐惧或误判。
3. **无代价**：故事中的错误没有后果，读者记不住教训。
4. **映射断裂**：故事中的某个关键设定在注解中没有对应的技术概念（即"为了文学性虚设情节"）。

---

## 三、技术审查清单

```markdown
### 技术审查

- [ ] **概念准确**：故事简化但没有歪曲技术原理。任何牺牲准确性的简化都在注解中明确标注。
- [ ] **完整映射**：注解中的 Key Mappings 表格覆盖了故事的所有核心设定。
- [ ] **映射精确**：每个"故事元素 → 技术概念"的对应关系是准确的，不是牵强附会。
- [ ] **术语规范**：使用领域内公认的中文/英文术语，不发明新词。
- [ ] **来源链接**：`sources` frontmatter 字段正确指向 wiki-web3 原文。
- [ ] **难度标注**：`difficulty` 与实际内容匹配（`beginner` = 科普级，`intermediate` = 需要基础理解，`advanced` = 面向从业者）。
```

### 技术审查的否决标准

1. **概念错误**：故事或注解中存在事实性技术错误（如把 `delegatecall` 说成 `call` 的行为）。
2. **危险简化**：简化方式可能误导读者形成错误理解，且未在注解中标注。
3. **映射缺失**：故事中的核心设定在 Key Mappings 中没有解释。
4. **来源错误**：`sources` 指向了不相关的 wiki 文章。

## 四、格式审查（CI 自动 + 人工抽查）

CI 会自动运行 `validate-fable.py`，以下问题会在 CI 中直接报错：

| 检查项 | 自动/人工 |
|--------|----------|
| frontmatter 必填字段完整 | CI 自动 |
| `category` / `difficulty` 枚举值合法 | CI 自动 |
| `## 故事` 章节存在 | CI 自动 |
| `## 这则寓言在说什么` 章节存在 | CI 自动 |
| Key Mappings 表格存在 | CI 自动 |
| 故事字数 800-6000 | CI 自动 |
| 正文无 Solidity 代码块 | CI 自动 |
| 文件名符合 `XX-slug.md` 规范 | CI 自动 |
| 双语标题（`title` + `title_cn`） | CI 自动 |

人工抽查项（每批 PR 随机抽 20%）：
- 链接是否可点击
- Markdown 渲染是否正常（表格、引用、分隔线）
- 目录 `concepts-catalog.md` 是否同步更新

## 五、评审话术模板

### 通过 (Approve)
```
叙事审查通过 ✅
隐喻选择贴切，延迟揭示节奏得当，角色动机清晰。

技术审查通过 ✅
映射准确，概念解释完整，难度标注合理。

建议可选（非阻塞）：第 3 段可以考虑增加一个细节来强化 XXX 的对应关系。
```

### 小修改 (Comment)
```
叙事审查需要小调整：
- [ ] 第 2 段过早提到了"合约"一词，建议改为更模糊的表达
- [ ] 结局的代价不够明确，建议让读者感受到 XXX 的直接损失

以上修改后可批准。
```

### 重大修改 (Request changes)
```
叙事审查 — Request changes

核心问题：故事的"中间态"隐喻与技术的"重入窗口"对应不够精确。
在故事里，出纳员每次都有"想销账但被打断"的心理活动，但技术上重入攻击利用的是
调用栈的递归特性，出纳员的"主观意愿"给读者造成了"如果出纳员再快一点就能避免"
的误解。建议重构：让攻击的本质表现为一种规则上的必然性，而非执行者的疏忽。

阻塞问题：
- [ ] 映射表缺少"排队仆人 → fallback 递归调用"的说明
- [ ] 注解中未提及 Checks-Effects-Interactions 模式

修改后可重新提交审查。
```

---

## 六、审阅者资质

- **叙事审查者**：至少已有一篇被合并的寓言，或被项目维护者（maintainer）认可。
- **技术审查者**：在对应领域有实际开发或研究经验，能独立判断技术准确性。

---

## 七、冲突解决

如果叙事审查与技术审查意见冲突（如叙事需要牺牲一点技术精确性），**默认以技术审查为准**，但可以在注解中通过 `"说明"` 列标注简化之处。如果双方无法达成一致，由项目 maintainer 裁决。

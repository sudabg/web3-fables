# The Numbered Lockers

> *"你以为锁在柜子里的秘密，其实柜子本身就在大厅中央。"*

## 故事

云州城的政务大厅里有一面巨大的墙，上面整齐排列着无数个编号柜——从 0 号到无限远。每一个柜子都能装下三十二个字，不多不少。

任何人都可以走到这面墙前，打开任何一个柜子，阅读里面的内容。没有任何门锁，没有任何密码。编号柜的设计哲学是：透明。

---

**柜子的规则**

大厅的管理员制定了一套严格的存取规则：

1. 每个柜子有固定的大小：三十二个字。如果一个东西小于三十二个字，可以把多个东西塞进同一个柜子。这叫"打包"。
2. 如果一个东西超过三十二个字，它必须占用连续的几个柜子。
3. 管理员按照你提交物品的顺序，依次放入 0 号、1 号、2 号柜子……
4. 一旦东西放进柜子，就不能移动。除非你主动把柜子清空，否则里面的内容永远在那里。

有一个精明的商人，他在柜子里存了自己的商业机密——他的成本价、他的客户名单、他的暗号。他以为锁在柜子里就安全了。

但他忘了：柜子没有门。任何人只要知道编号，就能打开来看。

有一天，一个竞争对手走到大厅，打开 3 号柜子——里面正是商人的成本价。打开 4 号柜子——客户名单。打开 5 号柜子——暗号。

商人崩溃了："这是我的隐私！"

管理员冷冷地回答："大厅的规则是透明。你放进来的时候，就应该知道这一点。"

---

**打包的艺术**

聪明的存储者学会了"打包"。

柜子的大小是三十二个字。如果你有两个各十六个字的东西，你可以把它们塞进同一个柜子——左边十六个，右边十六个。这样你只占用了一个柜子，而不是两个。

但打包有个微妙的限制：东西必须按大小从大到小排列。如果你先放了一个八位的东西，再试图放一个二十四位的东西——放不进去了，因为剩下的空间不够。但如果你先放二十四位的，再放八位的，刚好填满。

一个老练的存储顾问因此发了财。他的工作就是帮客户优化打包顺序，让同样多的东西占用最少的柜子。少占一个柜子，就少交一份存储税。

---

**动态柜子**

有些存储者需要存的东西数量不固定——比如一个不断增长的名单。管理员为他们设计了"动态柜子"：

- 名单的内容不直接放在编号柜里，而是放在另一个叫"大厅仓库"的地方。
- 编号柜里只放一个指针——一个地址，告诉你在哪里找到真正的名单。
- 当名单增长时，仓库会自动扩展。编号柜里的指针永远不变。

这种设计的代价是：每次读取名单，都需要先读编号柜里的指针，再去仓库找真正的内容。比直接存取要慢，但胜在灵活。

---

**清空柜子的陷阱**

有一个常见的误解：把柜子里的东西拿出来，柜子就空了。

不。在编号柜的世界里，"清空"意味着把柜子里的内容设置为零。但柜子本身还在，编号还在，它仍然占用着墙的空间，你仍然需要为它支付存储税。

唯一的例外是：如果你把柜子里的内容从零改回某个非零值，你不需要再付一次税——因为你已经为"创建"这个柜子付过了。但如果你创建了一个柜子，又把它清零，管理员会退还你一部分存储税，作为对你"释放空间"的奖励。

---

多年后，云州城来了一个外乡人，他看着满墙的编号柜，说了一句被刻在墙上的话："这不是存储，这是裸露。"

---

## 这则寓言在说什么

这则寓言对应的是 **EVM 的存储布局（Storage Layout）和 Slot 机制**。EVM 的存储是一个巨大的 key-value 映射，每个 slot 是 32 字节（256 位）。Solidity 按照变量声明顺序将数据 packing 到 slots 中，所有存储数据对链上所有人可见。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 编号柜墙 | EVM Storage | 每个合约有一个 `2^256` 个 slot 的存储空间，每个 slot 32 字节 |
| 三十二个字 | 32 bytes / 256 bits | 一个 storage slot 的固定大小 |
| 透明无锁 | 链上数据公开可读 | `storage` 中的任何数据都可以被任何人通过 `eth_getStorageAt` 读取 |
| 商业机密被读 | Storage 不是隐私 | 很多初学者误以为 `private` 变量是私密的，实际上只是 Solidity 层面限制访问，链上完全透明 |
| 打包 | Storage Packing | Solidity 尝试将多个小变量（如多个 `uint128`）pack 进同一个 slot，节省 gas |
| 打包顺序限制 | Packing rules | 变量按照声明顺序放入 slots。如果顺序不当（如先放 `uint8` 再放 `uint256`），会导致不必要的 slot 浪费 |
| 动态柜子 | 动态数组和映射 | `mapping` 和动态 `array` 使用 keccak256 哈希计算 slot 位置，数据存储在 "hash 空间" 而不是顺序 slots |
| 指针 | Slot 中的哈希值 | mapping 的 slot 中不存实际数据，只存计算偏移用的种子 |
| 清空柜子 | `delete` / 清零 | 将 slot 设为零会返还部分 gas（15,000 gas refund），但 slot 仍然存在于状态中 |
| 存储税 | SSTORE gas cost | 首次写入非零值消耗 20,000 gas；修改消耗 5,000 gas；清零返还 15,000 gas |

### 为什么是这个故事？

"链上数据公开"是 Web3 中最反直觉的事实之一。很多开发者误以为 `private` 变量是安全的，结果导致严重的安全漏洞（如著名的 `parity` 钱包漏洞）。

用"编号柜"来隐喻 storage，有三个直觉优势：
1. **透明性的可视化**：柜子没有门，任何人都能打开——对应了链上数据的完全公开
2. **打包的具象化**：把数据打包进固定大小的 slot，就像把东西塞进固定大小的柜子
3. **gas 成本的直觉**：创建柜子（SSTORE）贵，读取便宜，清空有退款——对应了 EVM 的 gas 定价逻辑

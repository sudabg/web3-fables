# The Prophet's Mold

> *"在东西被造出来之前，你就知道它会出现在哪里。"*

## 故事

云州城的铁匠铺有个传统：每当有人定制一把刀，铁匠会把刀打好，然后在刀柄上刻一个编号。编号是随机的——取决于当天打了第几把刀。买家只有在刀打好之后，才知道自己的刀是哪一把。

这个传统运转了三百年，直到一位叫沈算的年轻铁匠提出了一个大胆的设想。

---

**老办法：随机编号**

老办法的问题很明显：

1. 买家无法提前知道刀的确切编号。如果买家需要把刀和其他配件一起组装，他必须在刀打好之后才能开始组装。
2. 如果铁匠铺搬迁或者铁匠换人，编号序列可能中断，导致冲突。
3. 买家无法"预留"一个编号。如果他想确保自己的刀编号是 777，他只能祈祷当天恰好打到第 777 把。

---

**新办法：预言模具**

沈算设计了一种"预言模具"。这种模具的奇妙之处在于：在刀被铸造之前，你就能准确预言这把刀的最终编号。

预言的公式刻在模具本身上：

**刀的编号 = Hash(模具的编号 + 使用的材料配方 + 盐值)**

其中：
- 模具的编号是固定的——每个模具天生就有一个独一无二的编号。
- 材料配方是买家自己选择的——不同的配方产生不同的刀。
- 盐值是一个任意数字——买家可以自由选择，用来"调整"最终的编号。

这意味着：在刀被铸造之前，任何人都可以把这三个参数代入公式，计算出刀的编号。不需要等到刀打好，不需要信任铁匠，不需要任何预言。

---

**预言模具的应用**

这个发明在云州城掀起了一场革命。

**第一个应用：提前组装。**

一个刀鞘匠人需要为一把定制刀制作刀鞘。在老办法下，他必须等刀打好、知道编号之后，才能开始制作刀鞘——因为刀鞘必须和特定编号的刀匹配。

在新办法下，买家在下单时就知道刀的编号。刀鞘匠人可以在刀还在熔炉里的时候，就开始制作匹配的刀鞘。等刀打好，刀鞘也刚好完成。

**第二个应用：状态通道。**

两个商人需要在云州城设立一个"联合账户"来管理他们之间的往来账目。按照老办法，他们必须先设立账户，然后才能开始交易。

但在新办法下，他们可以在账户设立之前，就"预言"出账户的编号，然后基于这个预言编号签订合约："当我们最终设立账户时，账户编号将是 XYZ，届时合约自动生效。"

这个"账户尚未存在，但已经可以被引用"的状态，被称为"反事实"。

**第三个应用：批量铸造。**

一个军队需要一千把制式军刀。老办法下，铁匠一把一把地打，编号随机，无法预先分配。

用预言模具，军队可以在铸造开始前，为每一把刀预先计算编号，把编号提前刻在刀鞘上。铸造完成后，刀和刀鞘自动匹配。

---

**盐值的艺术**

预言模具引入了一个有趣的变量：盐值。

盐值本身没有任何意义——它唯一的作用是让买家可以"调整"最终编号。如果第一次计算出的编号已经被占用了，买家可以换一个盐值，重新计算，直到得到一个未被占用的编号。

这催生了"编号猎人"——一种专门寻找"漂亮编号"的职业。有人愿意花大价钱买一个编号里有八个 8 的刀，或者编号以特定数字结尾的刀。编号猎人用计算机不断尝试不同的盐值，寻找符合买家要求的编号。

---

多年后，沈算的预言模具成了云州城所有工匠的标准工具。有人问沈算："你的模具最厉害的地方是什么？"

沈算说："不是预言。而是它让'尚未存在的东西'变得'可寻址'。在刀被铸造之前，它就已经是一个可以被引用、被信任、被使用的实体。这改变了我们对'存在'的定义。"

---

## 这则寓言在说什么

这则寓言对应的是 **CREATE2**——EVM 中一种确定性的合约创建方式。与 CREATE 不同，CREATE2 的合约地址仅由 `deployer_address + salt + init_code_hash` 决定，与创建时的 nonce 无关。这意味着你可以在合约部署之前就精确预测它的地址。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 传统随机编号 | CREATE (基于 nonce) | 合约地址 = `keccak256(rlp([sender, nonce]))`。每次创建合约 nonce 增加，地址不可预测 |
| 预言模具 | CREATE2 | 合约地址 = `keccak256(0xff + sender + salt + keccak256(init_code))[12:]`。地址完全确定性，可预先计算 |
| 模具的编号 | Deployer 地址 | CREATE2 的计算中包含部署者地址，确保不同部署者的地址空间不冲突 |
| 材料配方 | Init code (初始化字节码) | 合约的 creation bytecode，决定合约的逻辑。不同的 init code 产生不同的地址 |
| 盐值 | Salt (bytes32) | 部署者可以自由选择的 32 字节值，用于在相同 init code 下生成不同地址 |
| 提前组装 | 预计算地址的应用 | 可以在合约部署前就知道其地址，提前配置其他合约引用它 |
| 反事实 | Counterfactual instantiation | 账户或合约尚未在链上部署，但已经可以被引用和交互（如状态通道、Layer 2） |
| 状态通道 | State Channels / Layer 2 | 基于反事实地址，可以在链下预先签订合约，链上仅在争议时实际部署 |
| 编号猎人 | Vanity address mining | 通过暴力枚举 salt 值，寻找特定前缀或模式的合约地址 |
| 批量铸造 | 批量部署 + 预计算 | 可以预先计算大量合约地址，批量配置和初始化 |

### 为什么是这个故事？

CREATE2 是一个看似简单、实则深刻的技术。它让"尚未部署的合约"获得了"可寻址性"，这是反事实状态、状态通道、批量部署等高级应用的基础。

用"预言模具"来隐喻 CREATE2，有几个直觉优势：
1. **确定性的可视化**：在铸造前就知道编号——对应了 CREATE2 的地址可预测性
2. **反事实的具象化**：刀还没打好，刀鞘已经可以开始做了——对应了 counterfactual instantiation
3. **盐值的灵活性**：任意调整盐值来寻找理想编号——对应了 salt 参数的作用

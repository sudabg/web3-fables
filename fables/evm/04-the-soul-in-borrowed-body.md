# The Soul in Borrowed Body

> *"借别人的身体说话，但话里的意思，全算在自己头上。"*

## 故事

云州城有一种古老的秘术叫"借身术"。施术者可以把自己的灵魂暂时移到另一个人的身体里，借用那个人的身体来行动。但借身术有一个极其危险的特性：

**你在借来的身体里做的一切，后果都由你原来的身体承担。**

---

**正常的访客**

正常情况下，如果一个人拜访另一个人，他站在对方家里说话。对方家里的陈设、家具、财物都是对方的。访客只是借用了一个空间来表达自己。

这叫"普通拜访"——访客有自己的身体，只是在对方的地盘上说话。访客说"我要喝水"，喝的是对方家里的水，与对方家里的财产发生交互。

**借身术的不同**

借身术完全不同。当甲把自己的灵魂移到乙的身体里时：

- 甲使用的是乙的身体、乙的喉咙、乙的四肢。
- 但甲说话的内容、做的决定、承担的后果，全都算在甲的头上。
- 如果甲用乙的身体打破了丙的窗户，丙要找甲赔偿，不是找乙。
- 如果甲用乙的身体签了一份契约，契约的效力绑定在甲身上，不是乙。

更关键的是：甲在乙的身体里行动时，他以为自己还在自己的家里。他看到的环境其实是乙的家——乙的家具、乙的财物、乙的日记。如果甲试图"把某件东西放进自己的保险箱"，他实际上是把东西放进了乙的保险箱。

---

**借身术的妙用：替身演员**

云州城的戏班子发现借身术有一个绝妙的用途：替身演员。

戏班子有一个主角叫阿龙。阿龙有极高的表演天赋，但他年事已高，无法亲自上台。戏班子找了一个年轻力壮的替身叫阿虎。

演出时，阿龙把自己的灵魂移到阿虎的身体里。观众看到的是阿虎的身体在表演，但每一个动作、每一句台词都是阿龙的灵魂在控制。阿虎只是一个人形躯壳。

阿龙可以随时"升级"自己的表演技巧——他在后台修炼新的招式，然后下次演出时，通过借身术让阿虎的身体执行新的招式。观众永远看到的是"阿虎"，但体验的是阿龙不断进化的艺术。

如果有一天阿龙去世了，戏班子只需要再找一个新的替身，阿龙的灵魂——如果提前备份过——可以继续借身演出。戏班子不需要重建整个舞台，只需要换一个新替身。

---

**借身术的灾难：灵魂错乱**

但借身术也是云州城历史上最危险的秘术。

有一次，一位术士在借身时犯了一个错误：他试图把一件东西"放进自己的口袋"，但他忘了自己正在借来的身体里。结果那件东西被放进了替身身体的口袋里——而这个口袋的位置，恰好对应着替身身体里一个存放剧毒的暗格。

剧毒泄露，替身当场死亡，术士的灵魂也被永远困在了死亡的躯壳里。

这个事故被称为"灵魂错位"——当术士以为自己在操作"自己的空间"时，实际上他在操作的是替身的空间，而两个空间的布局完全不同。

---

**借身术的规则**

经过无数次事故，云州城的术士们总结出了借身术的安全守则：

1. **替身身体的内部布局必须和原身体完全一致。** 如果原身体的"心脏位置"在左边，替身身体的"心脏位置"也必须在左边。否则灵魂会迷路。

2. **替身身体不能有自己的灵魂。** 如果替身身体里已经有一个灵魂，两个灵魂会争夺控制权，结果是灾难性的。

3. **借身期间，原身体处于休眠状态。** 原身体不能同时行动，否则会出现"两个我在同时做不同的事"的悖论。

4. **借身结束后，替身身体里不能留下任何秘密。** 因为灵魂离开时会带走记忆，但替身身体里留下的物理痕迹（如写在纸上的字）会暴露给下一个借身者。

---

## 这则寓言在说什么

这则寓言对应的是 **EVM 的 `delegatecall`**——一种特殊的低级调用方式。与普通的 `call` 不同，`delegatecall` 在**目标合约的代码**中执行，但使用**调用者合约的存储、余额和状态**。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 普通拜访 | CALL | 在目标合约的上下文中执行：使用目标的 storage、msg.sender 是调用者 |
| 借身术 | DELEGATECALL | 在目标合约的代码中执行，但使用调用者的 storage、msg.sender 和 msg.value 保持原样 |
| 甲的灵魂移到乙的身体 | 调用者用代理合约的代码执行 | 逻辑代码来自被调用合约（implementation），但 storage 是调用者（proxy）的 |
| 后果算在甲头上 | msg.sender 保持原样 | `delegatecall` 不改变 `msg.sender`，调用者身份保持不变 |
| 替身演员 | 代理合约模式 | Proxy 合约使用 `delegatecall` 将执行委托给 Implementation 合约，自己只负责存储状态 |
| 阿龙升级招式 | 可升级合约 | 更换 Implementation 合约地址，Proxy 的存储不变，逻辑升级 |
| 阿龙去世找新替身 | 合约迁移 | 旧 implementation 废弃，新 implementation 接入同一个 proxy，状态继承 |
| 灵魂错乱 / 剧毒暗格 | Storage collision | 如果 Implementation 和 Proxy 的 storage 布局不一致，`delegatecall` 会写入错误的位置，可能导致严重漏洞（如 Parity 多签钱包漏洞） |
| 布局必须一致 | Storage layout compatibility | 升级后的 Implementation 必须保持与旧版本相同的 storage 布局，新增的变量只能追加在末尾 |
| 替身不能有自己的灵魂 | 不能有状态冲突 | Implementation 合约不应该有自己的重要状态，所有状态应该在 Proxy 中 |
| 借身结束不留秘密 | 自毁风险 | 如果 Implementation 合约执行 `selfdestruct`，会通过 `delegatecall` 销毁 Proxy 合约（这是 delegatecall 的危险之一） |

### 为什么是这个故事？

`delegatecall` 是 Solidity 中最难以理解的概念之一，也是许多严重安全漏洞的根源（如 Parity 多签钱包被锁、各种代理合约漏洞）。

用"借身术"来隐喻 `delegatecall`，有几个直觉优势：
1. **上下文切换的可视化**：在别人的身体里，但后果算自己的——对应了 `delegatecall` 的上下文保持特性
2. **代理合约的直觉**：替身演员 = Proxy，阿龙 = Implementation，观众始终看到 Proxy 地址
3. **Storage collision 的危险**：灵魂错位 = storage 布局不匹配时的灾难性后果

# The New Grammar Manual

> *"旧的书写方式没有标点，新的手册把句子分成了章节。"*

## 故事

云州城的法律文书有一个延续了五百年的传统：所有的文书都是一卷连续的羊皮纸，从头到尾没有分段、没有标题、没有标点。法官必须从头读到尾，才能知道这份文书是契约、诉状还是遗嘱。

这个传统带来了三个问题。

---

**第一个问题：无法快速识别**

当一个信使把一卷新文书送到法院时，法官无法一眼看出这是什么类型的文书。他必须展开羊皮纸，从头开始读，直到发现某个关键词——比如" hereby "或" witness "——才能判断文书类型。

如果有人恶意把一份毒药配方伪装成契约（在前面写一段看起来像契约的开头，中间突然变成毒药配方），法官在读到一半时才会发现不对劲。但此时可能已经太晚了——某些行动已经基于前半段的误读而执行了。

---

**第二个问题：无法验证完整性**

羊皮纸可能会被虫蛀、被水浸、被裁切。如果文书的末尾缺了一段，法官很难发现——因为他不知道原本的文书应该有多长。他只能读到"突然中断"的地方，然后猜测：这是原本就这样结束的，还是后面缺了一段？

有些狡猾的讼师会故意删掉契约中对自己不利的最后几行，然后把残缺的契约呈上法庭。法官无法分辨这是原本就短的契约，还是被篡改过的长契约。

---

**第三个问题：执行效率低下**

法院的执行员（负责按照文书内容执行行动的人）必须从第一行开始，逐字逐句地读完整份文书。他不能跳过任何一段，因为不知道哪一段是相关的。

如果一份文书的前半部分是背景介绍（"甲方乃云州城居民，乙方乃东海商人……"），执行员必须把这些背景也读完，才能到达真正重要的执行条款。这极大地浪费了时间。

---

**新语法手册的革命**

一位叫章节的年轻法官提出了一套全新的文书格式："章节化文书"。

章节化文书不再是一卷连续的羊皮纸，而是被明确分成了几个部分：

1. **类型声明**（第一页）：明确写明"这是一份契约"或"这是一份诉状"。法官拿到文书的第一秒就知道该怎么处理。
2. **头部信息**（第二页）：列出文书的基本信息——作者、日期、版本。不需要展开全文就能获取元数据。
3. **正文**（第三页起）：真正的内容，分段清晰，每段有标题。
4. **校验页**（最后一页）：一段特殊的文字，通过对全文进行数学运算生成。如果文书的任何一页被改动，校验页就会不匹配，法官立刻知道文书被篡改过。

更关键的是：章节化文书在送入法院之前，必须先经过一个"格式审查员"的检查。审查员不看文书的内容，只看格式是否正确——类型声明是否在第一位、校验页是否完整、各章节是否有明确的分隔符。

如果格式审查不通过，文书根本不会被法官看到。这意味着：所有到达法官面前的文书，至少格式是正确的、结构是完整的、没有被截断的。

---

**意外的收益**

章节化文书推行后，法官们发现了一些意料之外的好处：

1. **执行变快了**。执行员不需要读完整份文书了。如果文书是"执行令"，他可以直接翻到"执行条款"章节，跳过前面的背景介绍。

2. **错误更容易发现**。如果一个讼师试图在契约里插入一段"自动转账"的条款，格式审查员会注意到这个条款不在契约允许的章节类型中，直接拒绝这份文书。

3. **版本兼容**。新格式的文书可以在旧法院里执行（旧法院仍然能读连续文本），但新法院可以充分利用章节化的优势。这是一个渐进式的升级，不是革命性的断裂。

---

多年后，云州城的所有法律文书都采用了章节化格式。有人问章节法官："你发明的最大贡献是什么？"

章节法官说："不是我发明了什么新内容。是我让文书有了'结构'。在旧格式里，内容和结构混在一起，无法分离。在新格式里，结构是可见的、可验证的、可优化的。一旦结构被显式化，很多事情就自然而然地变好了。"

---

## 这则寓言在说什么

这则寓言对应的是 **EVM Object Format (EOF)**——以太坊正在引入的一种新的字节码容器格式。EOF 将 EVM 字节码从"扁平的连续字节序列"转变为"结构化的容器"，包含类型声明、头部元数据、代码段、数据段等，并引入部署前的验证机制。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 连续羊皮纸 | 传统 EVM 字节码 | 扁平的字节序列，没有结构，EVM 逐字节解释执行 |
| 无法快速识别类型 | 无法区分 code/data | 传统字节码中，代码和数据混在一起，JUMP 目标需要运行时解析 |
| 毒药配方伪装成契约 | 恶意字节码 / 代码注入 | 可以在数据段中嵌入可执行代码，或通过 JUMP 跳转到任意位置执行非预期逻辑 |
| 无法验证完整性 | 无法静态验证 | 传统字节码无法在不执行的情况下验证其完整性和安全性 |
| 执行效率低下 | JUMP 解析开销 | EVM 需要在运行时计算 JUMP 目标，无法预先优化执行路径 |
| 章节化文书 | EOF 容器格式 | 字节码被组织为结构化的容器：magic number + version + header + code sections + data section |
| 类型声明 | EOF 头部类型信息 | 明确声明合约的代码段数量、数据段大小等元数据 |
| 校验页 | EOF 内置校验 | 容器格式的校验和/长度字段，确保字节码未被截断或篡改 |
| 格式审查员 | EOF 部署前验证 | EOF 合约在部署前必须通过静态验证，确保结构正确、JUMP 目标合法、栈深度安全等 |
| 格式不通过不上法庭 | 无效 EOF 无法部署 | 不符合 EOF 格式的字节码在部署时被拒绝 |
| 执行变快 | EOF 的性能优化 | 由于结构已知，EVM 可以预先编译和优化执行路径，不再需要运行时 JUMP 解析 |
| 渐进式升级 | EOF 与旧字节码共存 | EOF 是 opt-in 的，旧字节码继续兼容，新合约可以选择使用 EOF |

### 为什么是这个故事？

EOF 是 EVM 生态中长期且复杂的改进之一，涉及到底层执行模型的变革。对于不熟悉 EVM 内部的人来说，"字节码容器格式"听起来极其抽象。

用"章节化文书"来隐喻 EOF，有几个直觉优势：
1. **结构化的可视化**：从"一卷羊皮纸"到"分章节的文书"，让读者理解了"容器格式"的核心价值
2. **安全性的提升**：格式审查员在文书到达法官前拦截问题——对应了 EOF 的部署前验证
3. **性能优化的来源**：执行员直接翻到相关章节——对应了 EOF 预先知道代码结构后可以做的优化

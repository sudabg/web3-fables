# 传令筒 / The Message Tube

> *"只读不写的信筒，传给别人的话，自己不能改。"*

## 故事

云州城的通信系统有三种传递消息的方式。

---

**第一种：口信**

最古老的方式是直接口述。你走到对方面前，开口说话。对方听到什么就是什么。

口信的好处是简单直接。坏处是：如果你要告诉对方一百个人的名字，你得一个个念。对方记不住怎么办？只能再念一遍。

而且口信不能"传递"——你无法让朋友把你的口信再转达给第三个人。口信只在说话者和听者之间有效。

---

**第二种：书信**

你写一封信，装进信封，交给信使。信使把信送到对方手里，对方打开信封阅读。

书信可以传递复杂的内容——一百个人的名字可以写在纸上，不会遗漏。书信也可以被转交——收信人可以把原信再转给其他人。

但书信有个成本：写信需要纸和墨，送信需要信使的报酬。而且书信一旦被送出，里面的内容就不能改了。如果你想修改，必须重新写一封新的。

---

**第三种：传令筒**

云州城的军队使用一种特殊的通信工具：传令筒。

传令筒是一根密封的铜管，里面装着一张纸条。纸条上的内容在装入铜管时就固定了——任何人都可以把铜管打开，读出纸条上的内容，但没有人能修改它。

传令筒有三个独特的性质：

**第一，只读不写。**

收到传令筒的人只能读里面的内容，不能往里面加字、改字或删字。如果将军的命令是"前进十里"，传令筒里的纸条永远是这四个字。接收者不能把它改成"前进二十里"。

**第二，透明公开。**

传令筒没有锁。任何人路过都可以打开看里面的内容。这在军队里不是问题——命令本身不需要保密，只需要确保不被篡改。

**第三，一次性的。**

传令筒在送到目的地后，就完成任务了。里面的纸条可以被抄录存档，但铜管本身不会被 reused。下次传令，用一根新的铜管。

---

**传令筒的妙用：远程指挥**

云州城的将军驻扎在北疆，皇帝在京城。皇帝想命令将军做一件事，但他不能亲自去北疆口述。他可以写一封书信，但书信在路途上可能被篡改、被截获。

传令筒解决了这个问题：皇帝把命令写成纸条，装入铜管，密封。信使运送的是铜管，不是裸信。沿途任何人都可以打开铜管看命令的内容（透明），但没有人能在不破坏密封的情况下修改命令（不可篡改）。

将军收到铜管后，打开它，读取命令，执行。执行过程中，将军可以对照铜管里的命令，确认自己没有理解错。

---

**传令筒的限制**

但传令筒也有它的局限。

如果皇帝想让将军"根据前线情况灵活决定"，传令筒就不够用了——因为传令筒里的命令是固定的，将军无法向皇帝实时请示。他只能在收到传令筒后，按照既定命令执行。

如果命令中包含了大量数据——比如"这里有一份五百人的名单，请按名单发粮"——传令筒可以装下这份名单（因为它是一张纸），但将军如果需要把这些名字一个个输入到账本里，他就得手动抄写。传令筒里的数据不能直接"导入"到账本，只能被阅读。

---

## 这则寓言在说什么

这则寓言对应的是 **EVM 的 Calldata**——交易或消息调用中携带的只读数据。Calldata 是外部调用时传入的数据，存储在交易体中，不能被被调用合约修改，只能通过 `msg.data` 或函数参数读取。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 口信 | 内部调用 / 本地计算 | 合约内部的函数调用，数据在栈和内存中传递，不需要序列化 |
| 书信 | Memory / Storage | 可写的数据存储区域，可以被修改和传递 |
| 传令筒 | Calldata | 外部调用的只读数据，随交易传递，被调用合约只能读取不能修改 |
| 只读不写 | Calldata 的不可变性 | `calldata` 位置的数据不能被修改。Solidity 中标记为 `calldata` 的参数是只读的 |
| 透明公开 | Calldata 的公开性 | 交易数据对全网公开，任何人可以通过区块链浏览器读取 |
| 一次性 | Calldata 的生命周期 | Calldata 只在当前消息调用期间存在，调用结束后不可访问 |
| 密封铜管 | 交易的完整性 | 交易被签名后，calldata 不能被篡改（否则签名验证失败） |
| 远程指挥 | 跨合约调用 | 合约 A 调用合约 B 时，通过 calldata 传递函数选择器和参数 |
| 大量数据装入铜管 | Calldata 的容量 | Calldata 可以携带较大的数据（如数组、字符串），但读取和解析需要 gas |
| 手动抄写 | Calldata 到 Memory/Storage 的复制 | 如果需要修改 calldata 中的数据，必须先复制到 memory 或 storage |
| 无法实时请示 | Calldata 的单向性 | 一次调用中，calldata 是单向的。被调用者无法修改 calldata 并返回给调用者 |

### 为什么是这个故事？

Calldata 是 EVM 中三个主要数据位置（storage、memory、calldata）之一，但它的"只读"和"外部"特性让很多初学者困惑。

用"传令筒"来隐喻 calldata，有几个直觉优势：
1. **只读性**：传令筒只能读不能改——对应了 calldata 的不可变性
2. **外部性**：传令筒用于远程通信——对应了 calldata 只存在于外部调用中
3. **透明性**：任何人可以打开看——对应了链上交易的公开性

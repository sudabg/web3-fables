# The Runestones

> *"每一块石头上只刻一个字，但按照正确的顺序排列，它们能建造一座城。"*

## 故事

云州城的建筑工地上不使用图纸。工匠们使用一种古老的系统：符文石。

每一块符文石是一个手掌大小的石块，上面刻着一个符号。符号的种类很少，只有大约两百种。但这两百种符号，按照不同的顺序排列，可以表达任何建筑的建造方法。

---

**符文石的工作方式**

建筑工地中央有一个大石台，石台上只能同时摆放三十二块符文石。这是石台的物理限制——再多就放不下了。

工匠的工作流程是这样的：

1. 他从仓库取出符文石，按顺序放到石台上。
2. 当石台上的符文石达到某个操作所需的数量时，自动触发一个动作。
3. 动作完成后，石台上的符文石被替换为动作的结果。
4. 他继续放新的符文石，触发下一个动作。

例如：
- 操作"加"：需要石台上有两块符文石（比如"3"和"5"）。触发后，两块石头被移除，换成一块新石头"8"。
- 操作"复制"：需要石台上有一块石头。触发后，石头被复制一份，石台上现在有两块相同的石头。
- 操作"交换"：需要石台上至少有两块石头。触发后，最上面的两块石头交换位置。

---

**符文石的翻译**

高级建筑师不会直接和符文石打交道。他们用一种叫"蓝图语言"的高级语言来设计建筑——"在这里建一道墙，墙高十尺，宽两尺"。

然后，一位专门的"翻译师"把蓝图语言转换成符文石的序列。这个过程是确定性的：同样的蓝图总是产生同样的符文石序列。

但翻译不是一一对应的。蓝图上的一句"墙高十尺"，可能被翻译成几十块符文石的复杂操作——计算面积、检查承重、分配材料。翻译师在翻译时还会做优化：如果蓝图上连续出现了三次"墙高十尺"，翻译师不会重复翻译三次，而是把结果缓存起来，重复使用。

---

**符文石的直接操控**

有些极其精密的建筑，蓝图语言无法精确表达。这时候，大师级工匠会直接编写符文石序列。

直接操控符文石的优势是：
- **精确**：每一颗石头的位置都在工匠的控制之下，没有翻译的模糊性。
- **高效**：翻译师为了安全会加入很多检查，但直接操控可以跳过不必要的检查，节省时间和材料。
- **访问隐藏功能**：蓝图语言只暴露了常见操作，但有些特殊操作只有直接操控符文石才能使用。

风险也很明显：
- 如果工匠把两块石头的顺序搞错了，整座建筑可能倒塌。
- 如果工匠忘了在某一步之后清理石台，石台被占满后，后续操作无法进行。
- 如果工匠使用了"自毁"符文——一块特殊的、能让整个工地消失的石头——后果是不可逆的。

---

**符文石的不可变性**

一旦符文石序列被刻好、被送到工地，就不能再改了。如果建筑师发现设计有缺陷，唯一的办法是刻一套新的符文石序列，废弃旧的。

这意味着：符文石的编写必须极其谨慎。一个小错误——比如把"加"和"减"搞混了——可能导致完全错误的结果，而且无法在现场修正。

---

多年后，云州城最高的塔楼是用符文石建造的。塔楼的设计者是一位直接操控符文石的大师。他说："蓝图语言让普通人也能建房子。但如果你要建一座塔楼触及云端，你必须理解每一块石头的意义。"

---

## 这则寓言在说什么

这则寓言对应的是 **EVM 的操作码（Opcodes）和字节码（Bytecode）**。EVM 是一个基于栈的虚拟机，所有智能合约最终都被编译成字节码——一系列操作码的序列。Solidity 是高级语言，编译器将其翻译成 EVM 字节码。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 符文石 | EVM 字节码 / 操作码 | 每个 opcode 是一个字节（或更多），EVM 逐条解释执行 |
| 约两百种符号 | ~150 个有效 Opcodes | EVM 定义了约 150 个操作码，涵盖算术、逻辑、存储、调用、环境信息等 |
| 石台放三十二块石头 | EVM 栈深度限制 | EVM 栈最大深度为 1024 个 256-bit 字。超出会导致栈溢出 |
| 操作"加" | ADD opcode | 从栈顶弹出两个值，相加后压回栈顶。所有 EVM 计算都在栈上进行 |
| 操作"复制" | DUP opcode | 复制栈中的某个值到栈顶 |
| 操作"交换" | SWAP opcode | 交换栈顶两个值的位置 |
| 蓝图语言 | Solidity / Vyper 等高级语言 | 开发者用高级语言编写合约，编译器翻译成字节码 |
| 翻译师 | Solidity 编译器 (solc) | 将高级语言编译成 EVM 字节码，并进行优化 |
| 翻译优化 | 编译器优化 | 常量折叠、死代码消除、循环优化等 |
| 直接操控符文石 | 内联汇编 (Yul / Inline Assembly) | 开发者可以直接编写 opcode 序列，获得更精确的控制和更高的效率 |
| 访问隐藏功能 | Assembly 访问底层 | 某些 EVM 特性（如 `create2` 的精细控制、内存布局优化）只能通过汇编访问 |
| 石台被占满 | Stack too deep error | Solidity 函数中局部变量过多时，栈深度超过 16 个槽位的限制（编译器层面的限制） |
| 自毁符文 | SELFDESTRUCT opcode | 销毁合约，发送余额到指定地址，不可逆 |
| 刻好后不能改 | 已部署字节码不可变 | 合约部署后，代码不可修改（除非使用代理合约模式） |

### 为什么是这个故事？

EVM 字节码和操作码是智能合约的"最终形态"，但大多数开发者只写 Solidity，从未直接看过字节码。理解"高级语言下面是什么"对于调试、优化和安全审计至关重要。

用"符文石"来隐喻字节码，有几个直觉优势：
1. **栈机器的具象化**：石台 = 栈，放石头 = push，操作 = pop 和计算
2. **编译过程的可视化**：蓝图 → 翻译师 → 符文石 = Solidity → solc → bytecode
3. **Assembly 的价值**：直接操控符文石 = 内联汇编，精确但危险

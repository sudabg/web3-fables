# The Infinite Borrower

> *"规则写得太信任人，本身就是一种漏洞。"*

## 故事

洛城最大的金库叫恒金库，由城主亲自立法保护。恒金库的规矩写在青铜板上，一字不改：

**任何人随时可取回自己存入的全部黄金。取金流程分两步：先交付，后销账。**

这规矩听起来公平至极。出纳员阿仓在此工作二十年，从未出错。他的动作像钟表一样精确：顾客递上存票，他转身从库里搬出黄金，双手奉上，然后才翻开账本，划去那笔存款。

"先交付，后销账"——这是城主的仁慈。他说："取金的人本就急着用钱，怎能让他们等着我慢慢记账？"

城里有位商人叫魏狡，以精明著称，但没人说他不守规矩。他确实守规矩——他守的是青铜板上写的规矩，一字不差。

魏狡在恒金库存了一百两黄金。某天清晨，他派自己的管家老周去取金。

老周站在柜台前，递上存票："取一百两。"

阿仓验过存票无误，转身进库，搬出一百两黄金，放在柜台上。黄金很重，老周没有立即离开。他从袖中取出一张纸，写上新的取金请求，折好，交给门口一个等候多时的信童。信童飞奔回恒金库的正门，再次排进队伍里。

阿仓刚要翻开账本销账，抬头一看，柜台前又站着一个人——还是老周。不对，是另一个老周？不，是魏狡家的另一个仆人，穿着同样的衣服，递上一张存票："取一百两。"

阿仓皱眉："魏狡的存款不是刚刚才取吗？"

仆人答："大人还有另一笔存款，这是另一张存票。"

阿仓查验——存票是真的，魏狡确实还有一百两在库中。阿仓只好转身再进库，再搬一百两。

就在阿仓放下第二堆黄金的瞬间，第三个仆人已经站在柜台前了。

阿仓开始冒汗。他每次都想先销账，但每次都被人抢先一步站在柜台前。城主的规矩说得很清楚：顾客在柜台前，就必须先服务。阿仓不能对柜台上的人说"你等一下，我先记账"。那会被视为怠慢。

第四波、第五波、第六波……魏狡的仆人像流水一样涌来。恒金库的黄金在减少，账本上的数字却纹丝未动——因为阿仓从未有机会翻开它。

直到库里传来一声空洞的回响。最后一块金砖被搬走了。

阿仓瘫坐在椅子上，看着空荡荡的库房门，终于翻开了账本。账本上清清楚楚地写着：魏狡，存款余额一千两。

而魏狡总共只存过一百两。

---

城主震怒，召见阿仓和全城智囊。一位年轻的幕僚叫林守，他仔细研究了青铜板上的规矩，发现问题不在阿仓，也不在魏狡——魏狡确实没有违反任何明文规定。

"问题在顺序，"林守说，"先交付，后销账。在交付和销账之间的那一瞬间，金库处于一种危险的中间态：黄金已经出去了，但账本上它还属于金库。任何人只要在这个瞬间再次请求取同一笔钱，金库就会重复支付。"

"那改成先销账，后交付呢？"

"可以解决大部分问题，但还有更彻底的办法，"林守说，"给金库加一把'进程锁'。当一次取金流程开始时，金库进入忙碌状态，同一笔存款的任何新请求都被暂时拒绝，直到流程完全结束。这样，即使顺序不变，中间态也不会被利用。"

城主采纳了两种改进：所有金库改为先销账后交付，同时在高风险操作中强制使用"进程锁"。

魏狡后来怎样了？据说他带着黄金去了邻国，开了一家更大的金库。没人知道他有没有把那个漏洞写进自己的规矩里。

## 这则寓言在说什么

这则寓言讲的是智能合约中最著名的安全漏洞之一：**重入攻击（Reentrancy Attack）**。

2016 年，以太坊上的 The DAO 项目因此漏洞被窃取了价值约 6000 万美元的 ETH，直接导致了以太坊的硬分叉和以太坊经典（ETC）的诞生。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 恒金库 | 智能合约（如 EtherStore） | 运行在区块链上的自动化程序，规则由代码定义 |
| 青铜板规矩 | 合约代码逻辑 | "先交付，后销账"对应 `withdraw()` 函数中先执行外部调用（`call`），再更新余额（`balances[msg.sender] = 0`） |
| 阿仓 | 合约的执行环境（EVM） | EVM 按代码顺序逐条执行，不会在执行中途自动拒绝外部调用 |
| 魏狡的管家与仆人 | 攻击合约（Attacker Contract） | 攻击合约在接收 ETH 时触发 `fallback()` / `receive()` 函数，在其中再次调用目标合约的 `withdraw()` |
| 信童与排队仆人 | 递归调用链 | 每次外部调用都会重新进入 `withdraw()`，形成递归循环，直到金库耗尽 |
| "中间态" | 合约的重入窗口（Reentrancy Window） | 合约状态在 `call` 之后、`balances` 更新之前与预期不一致，这种状态不一致性是漏洞的本质 |
| 先销账后交付 | Checks-Effects-Interactions 模式 | Solidity 安全编码规范：先检查条件（Checks），再更新状态（Effects），最后才进行外部交互（Interactions） |
| "进程锁" | 重入锁修饰器（noReentrancy / mutex） | 使用布尔变量 `locked` 或 OpenZeppelin 的 `ReentrancyGuard` 确保函数执行期间不可重入 |

### 为什么是这个故事？

重入攻击的核心是**时间上的重叠**——一次操作尚未完成，另一次操作就插了进来。用"管家在出纳员记账之前派新仆人排进队伍"来隐喻这个时序问题，比直接说"递归调用"更直觉。

故事刻意让魏狡"没有违反任何明文规定"，因为重入攻击的可怕之处就在于：**攻击者没有伪造签名、没有越权访问，他们只是以合约允许的方式，以合约没预料到的顺序，反复调用同一个函数。** 攻击者遵守规则，但规则的顺序写错了。

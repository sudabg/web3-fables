# The Seven Gates

> *"每扇门都通向同一个宝藏，但每扇门后都站着不同的守卫，各有各的疏忽。"*

## 故事

风语国有一座藏宝库，由七道门守护。每道门都有专门的守卫和独特的验证方式。国王相信，七重保险足以让任何盗贼束手无策。但事实证明，七道门只是七种不同风格的疏忽。

---

**第一道门：无人看守的后门**

第一道门是宝库的主入口，装饰着最华丽的铜钉。但守卫阿大有个习惯——他总认为"只有国王的人才敢靠近主门"，所以他从不认真核对来人的身份。任何人只要穿着像模像样的衣服，大摇大摆地走进去，阿大连头都不抬。

有个小偷发现了这一点。他没有撬锁，没有翻墙，只是换了一件仿制的官服，径直穿过了第一道门。阿大还以为他是新来的内侍。

**第二道门：信任陌生人的引路人**

第二道门的守卫阿二是个热心肠。他的职责是核对来人的引路文书——只有持有国王亲签文书的人才能继续深入。但阿二嫌麻烦，他发明了一个"快捷方式"：如果来人声称"我是某某大人派来的"，阿二就不再索要文书，而是直接放行。

一个骗子冒充外地使节，编了一个听着像那么回事的头衔。阿二被唬住了，恭恭敬敬地让开了路。

**第三道门：无限的回旋**

第三道门的守卫阿三是个死脑筋，严格按照规矩办事。规矩是：来访者递上请求单，阿三核实后从库里取物，交给来访者，然后销去记录。

一个聪明的骗子利用了规矩的顺序。他递上请求单，阿三转身去取物。就在阿三把东西递出来的瞬间，骗子没有离开，而是再次递上一张同样的请求单。阿三很困惑，但规矩没说"不能连续请求"，所以他转身再去取物。就这样，阿三在同一笔记录被销掉之前，反复取出了十份同样的宝物。

**第四道门：算盘溢出**

第四道门的守卫阿四掌管着数量登记。他有一把老算盘，能显示的数字最大到九十九万九千九百九十九。国王觉得这已经够大了——谁会有超过一百万的东西呢？

一个骗子研究了很久，发现了一件可怕的事：如果算盘上显示的数字已经是九十九万，再往上加二十万，算盘不会显示"一百一十九万"（因为它显示不了），而是会跳回零，从头开始计数。

骗子精心设计了请求：他要取走的宝物，按数量登记，恰好能让算盘溢出归零。当算盘归零时，系统认为他取走了零件东西——他的余额纹丝未动，但库里确实少了那么多宝物。

**第五道门：疲于奔命的信使**

第五道门的守卫阿五负责向所有有份额的人分发红利。他的方式是：拿到名单，一个一个按顺序敲门送达。名单越长，他跑得越累。

一个骗子发现了这个规律。他注册了成百上千个假身份，把自己塞进了名单里。当阿五拿着超长的名单开始奔跑时，还没跑完一半就已经精疲力竭，瘫倒在半路。整个分发系统因此瘫痪，而骗子注册的那些假身份里，有几个恰好插在了名单的前端，提前拿到了红利。

**第六道门：过时的天气预报**

第六道门的守卫阿六有个特殊职责：他根据城外的气象塔来决定今天开放哪些库房。如果气象塔报告"暴雨"，他就开放防水物资库；如果报告"干旱"，他就开放水源库。

气象塔由一位老气象员维护。老气象员每天只更新一次数据，而且有时会因为喝酒而延误。一个骗子买通了气象塔下的信童，让他故意传递三天前的旧报告。当阿六根据过时的"暴雨"报告打开防水物资库时，真正的需求是抗旱——城里渴得要命，而防水物资堆积如山。

更糟的是，骗子还在气象塔上动了手脚。他在塔顶安装了一面可以旋转的镜子，在特定时刻把阳光反射到温度传感器上，让气象塔"测到"异常高温。阿六根据这个伪造的数据，做出了完全错误的决策。

**第七道门：所有门的总结**

第七道门没有守卫。它是一面镜子，让每个通过前六道门的人看看自己是怎么进来的。

国王最终发现失窃时，召集了七位守卫问责。阿大低头说："我以为没人敢冒充。"阿二说："我以为头衔不会骗人。"阿三说："我按规矩办了，规矩没说要防着重复请求。"阿四说："算盘是祖传的，我没想到有人会算到那么大。"阿五说："名单太长不是我的错。"阿六说："气象塔的数据我能怎么办？"

国王沉默了很久，最后说："你们的错误各不相同，但根源只有一个——你们每个人都假设了'不会有人这么做'。盗贼最擅长的，就是找到你们假设的裂缝。"

后来，国王请来了风语国最挑剔的安全师。安全师没有换掉任何一道门，而是做了一件事：在每一道门旁边挂了一块牌子，上面写着——

"不要假设好人会来。要假设最坏的人已经站在门外。"

## 这则寓言在说什么

这则寓言系统性地对应了 Solidity 智能合约中最常见的六类安全漏洞（第七道门是总结）。原文 wiki 概念覆盖了访问控制失效、未检查外部调用、重入攻击、整数溢出/下溢、Gas 耗尽、Oracle 数据过时和操纵等问题。

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 第一道门无人看守 | 访问控制失效（Access Control Failure） | 未使用 `onlyOwner` 等修饰符保护敏感函数，任何人都可以调用特权操作 |
| 第二道门信任陌生人 | 未检查外部调用（Unchecked External Calls） | 信任任意合约交互，未验证调用方身份或未使用白名单机制 |
| 第三道门无限回旋 | 重入攻击（Reentrancy） | 外部调用先于状态更新，攻击者利用回调函数递归调用同一函数 |
| 第四道门算盘溢出 | 整数溢出/下溢（Integer Overflow/Underflow） | Solidity <0.8.0 中 `uint256` 运算溢出会静默回绕。BEC 事件中 `cnt * _value` 溢出为 0，导致无成本增发 |
| 第五道门疲于奔命 | Gas 耗尽 / DoS（Gas Exhaustion） | 遍历所有用户的推送模式（push）在用户数过多时会超出 gas limit，导致功能瘫痪。应使用拉取模式（pull） |
| 第六道门过时天气 | Oracle 数据陈旧与操纵（Oracle Manipulation） | 依赖单一、不更新的链下数据源，且未做异常值检查和 staleness 检查。AMM 价格预言机可被闪电贷操纵 |
| 第七道门镜子 | 安全审计思维 | 所有漏洞的共同根源是开发者的"善意假设"。安全审计的核心就是系统性打破这些假设 |
| 安全师的牌子 | 对抗性思维（Adversarial Thinking） | 以攻击者视角审视代码，假设最坏情况已经发生 |

### 为什么是这个故事？

用"七道门"来隐喻多种安全漏洞，有两个好处：

1. **系统性**：读者可以直观地看到，安全不是一个单一的问题，而是一个系统工程。每道门的守卫都"按自己的方式"犯了错，但组合起来就是灾难。
2. **归因明确**：每个守卫的借口（"规矩没说""算盘是祖传的""数据我能怎么办"）恰好对应了开发者的典型推诿——"代码就是这么写的""Solidity 旧版本的问题""Oracle 的数据问题"。国王的最后一句话点出了核心：安全是设计者的责任，不是工具的借口。

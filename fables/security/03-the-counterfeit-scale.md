# 伪造的秤 / The Counterfeit Scale

> *"最危险的谎言，是机器替你撒的。"*

## 故事

青国最大的钱庄叫万通号，专门帮商人批量转账。万通号的规矩是这样的：一位商人如果要把银子分给二十个伙计，只需写一张单子，注明每人多少两，钱庄的账房先生就会一次性把二十笔账全部算清。

账房先生的算盘是从前朝传下来的，极其精美，拨珠如飞。但有一个不为人知的秘密：这把算盘最多只能显示九十九万九千九百九十九两。超过这个数，算盘不会报错，而是像车轮一样——从九十九万九千九百九十九再拨一格，就会归零，从零重新开始计数。

没有人知道这个秘密。算盘用了三十年，从未出过问题。因为正常的生意里，谁会一次性转超过一百万两银子呢？

---

商人钱多的绰号叫"钱不倒"。他不是万通号的老客户，但出手阔绰，出手就是一千两的见面礼。

钱多不是来做正常生意的。他花了三个月研究万通号的账簿，发现了那把算盘的秘密。他算出了一个精确的数字——

如果他要给两个人各转五十万两，总计就是一百万两。而算盘的极限是九十九万九千九百九十九两。一百万两，刚好溢出归零。

归零意味着什么？意味着账房先生的算盘上显示的是"零"。而零，在万通号的系统里有一个特殊的含义："无需扣款"。

钱多设计了一个完美的请求：

他要给两个伙计各发一笔"奖金"。每个伙计的奖金数额，恰好是五十万两。

账房先生接过单子，习惯性地开始计算：2 人 × 50 万两 = 总计 100 万两。

算盘响了——咔嗒。显示：零。

账房先生愣了一下，但很快释然。零意味着无需从钱不倒的账户扣款。规矩写得很清楚："若总计为零，则视为无扣款交易。"

于是他照办了。两个伙计各收到了五十万两银子，而钱不倒的账户分文未动。

钱多没有停手。他发现了一个更惊人的事实：既然"零"意味着"无需扣款"，那他可以把所有伙计都拉进来。二十个伙计，每人五十万两。算盘上还是零。因为 20 × 50 万 = 一千万两，远超算盘的极限，仍然溢出归零。

账房先生一如既往地机械执行。二十个假伙计，每人五十万两，总计一千万两——算盘显示：零。无需扣款。

钱多没有贪心到榨干万通号。他拿了足够跑路的银子，连夜离开了青国。而万通号直到三天后对账才发现：库里的银子少了整整一亿两，但账本上没有任何一笔对应的扣款记录。

---

钱庄的东家疯了。他找来了全国最厉害的算师查账。算师花了七天七夜，终于发现了那把算盘的秘密。

"问题不在账房先生，"算师说，"账房先生每一步都是按规矩执行的。问题在于规矩本身——它假设了'总计不可能为零'，而'零'被定义为'无需扣款'。当溢出导致总计归零时，这个假设就被打破了。"

"那把破算盘为什么不报错？"

"因为它太老了。老算盘的设计哲学是'不中断、继续算'。它不会告诉你说'数字太大了'，它只是默默地从头开始。"

"那现在怎么办？"

"两种办法。第一，换一把新算盘——新算盘会在数字过大时直接拒绝计算，而不是悄悄归零。第二，在每一次计算之前，先检查一下：如果乘积除以其中一个乘数，结果是否等于另一个乘数？如果不等，说明计算过程中发生了溢出。"

东家采纳了两种改进：全面更换新算盘，同时在每次批量转账前增加溢出校验。

后来有人在邻国遇到了钱多。问他为什么不拿更多。钱多笑了笑说："够用就好。而且，如果我拿得太多，那个算盘的秘密就会提前暴露。我要的不是一次性的暴利，而是让那个秘密一直存在——这样我下次还能用。"

---

## 这则寓言在说什么

这则寓言对应的是 2018 年 BEC（美链）Token 的**整数溢出攻击**，该漏洞导致一个市值一度突破 280 亿美金的项目归零。漏洞仅存在于一行代码：`uint256 amount = uint256(cnt) * _value;`

### 关键映射

| 故事元素 | 技术概念 | 说明 |
|---------|---------|------|
| 万通号 | BEC Token 合约 | 批量转账功能的智能合约 |
| 账房先生的老算盘 | Solidity <0.8.0 的 uint256 | `uint256` 最大值为 `2^256 - 1`。当运算结果超过此值时，低 256 位被保留，高位被截断，表现为"静默回绕"到零 |
| 算盘的九十九万极限 | `uint256` 的最大值 | `115792089237316195423570985008687907853269984665640564039457584007913129639935` |
| 钱多设计的 2 × 50 万 = 100 万 | 攻击参数 | 攻击者传入 `cnt=2`，`_value=57896044618658097711785492504343953926634992332820282019728792003956564819968`，乘积溢出为 0 |
| "零意味着无需扣款" | `require(balances[msg.sender] >= amount)` | 当 `amount` 溢出为 0 时，任何余额都满足 `>= 0`，扣款逻辑被跳过，但转账仍然执行 |
| 新算盘 | Solidity 0.8.0+ | 0.8.0 之后编译器默认启用溢出检查，溢出会触发 revert |
| 溢出校验 | `assert(amount / _value == cnt)` | 乘法后做除法回验，如果结果不一致说明发生了溢出 |
| SafeMath | OpenZeppelin SafeMath 库 | 在 <0.8.0 版本中，使用 `.mul()` 代替 `*` 来强制检查溢出 |

### 为什么是这个故事？

整数溢出是编程中最古老的漏洞之一，但它在区块链上的破坏力被放大了——因为智能合约处理的是真实的资产，且代码不可更改。

用"算盘溢出归零"来隐喻这个漏洞，有三个直觉上的优势：

1. **可视化**：算盘是一个物理对象，"归零"是可以想象的动作。
2. **非恶意性**：账房先生不是坏人，他只是"按规矩执行"——这恰好是智能合约的特性：代码不会"故意"作恶，它只是忠实地执行有缺陷的逻辑。
3. **攻击的精巧性**：钱多的攻击不是暴力破解，而是对系统边界的精确计算。这和真实攻击者花费三个月研究 `uint256` 边界的行为一致。
